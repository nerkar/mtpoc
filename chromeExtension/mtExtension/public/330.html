If you’re using external security and you’re using the Abend-AID Web Browser
Interface viewing access method, you may receive an error message if you don’t

Specifying the User Identifier to BDCAS, TDCAS, and Viewing Server
Depending on whether you start the BDCAS, TDCAS, and viewing server as a job or a
started task, perform one of the procedures below to specify the user ID for each address
space.
If You Start the BDCAS, TDCAS, or Viewing Server as a Job
If you start the Abend-AID address space as a job, include the parameters USER= and
PASSWORD= on the job statement. For example:
//ADDSPACE JOB
.....
,USER=AAADSPAC,PASSWORD=password,
In the above example, AAADSPAC is the user ID of a particular BDCAS, TDCAS, or Abend-
AID viewing server address space.
If You Start the BDCAS, TDCAS, or Viewing Server as a Started Task
The method you use to define the user identifier to the Abend-AID address space differs,
depending on what external security package you are using.
With RACF
For RACF release 2.1 and more current, if you start the Abend-AID address space as a
started task, define a dynamic started procedure for the address space to RACF using the
following command:
RDEFINE STARTED proclib_member_name.jobname
Alternatively, if you run the Abend-AID address space as a started task and you use RACF,
you can associate the cataloged procedure name of the address space with a suitably
authorized RACF user through the now deprecated RACF table, ICHRIN03, which is
resident in the link pack area (LPA). RACF supplies a default ICHRIN03 table, which you
can modify.
With ACF2
If you use ACF2 and you start the Abend-AID address space as a started task, define the
cataloged procedure name of the address space as a valid ACF2 user.
With Top Secret
If you use Top Secret and you start the Abend-AID address space as a started task, define
the cataloged procedure name of the address space to Top Secret using the following
command:
TSS ADD(STC)
....
